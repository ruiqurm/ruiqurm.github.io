Ubuntu lldb-mi not found

发表于 2021-05-09 更新于 2021-05-10

最近把gcc换成clang，调试的时候vscode提示lldb-mi不存在。

我顺着路径(/usr/bin/lldb-mi)去看了一下，发现根本就没有lldb-mi-10的文件。后来搜了一下才发现lldb-10不再顺便编译出lldb-mi，需要我们自己去编译。在一个issue中，一个大佬([@Wing Chau](https://github.com/iamWing))直接给了解决方案。我把它翻译下来整理如下：

  1. 安装相关的包
  1. 设置软链接。如果之前装过这些，前三个会提示已存在。
  1. 编译lldb-mi。同样，如果之前装过，/usr/bin/下会有一个lldb-mi。需要手动删掉

参考：https://github.com/microsoft/vscode-cpptools/issues/5415#issuecomment-662603085

Efficient Cpp

发表于 2021-04-25

这本书强调了几个问题来提高性能。

c++的性能是极好的，但是因为引入了类等机制，和纯C还是有的差距。这本书的目标就在于补足这段差距。

类的构造函数和析构函数

  • 构造函数和析构函数都很高效，但是它们常常包含冗余计算或者超前计算（比如一个值不一定会用到，可以等用到再算）
  • 不要滥用继承，递归的析构可能会使程序性能下降
  • 黑盒测试很好；但是对于性能要求较高的代码，我们需要对源代码进行更深入的测试
  • 不要随意创建对象（比如传参时，返回值时）。或者用引用，std::move，拷贝构造等机制规避。
  • 初始化的速度比初始化后再赋值要快。

虚函数

虽然虚函数很好，但是动态绑定的开销也是不容小觑的。并且，虚函数（在指针使用时）不能内联。

如果需要优化虚函数的话：

  • 模板（最好）

    和继承类似，但是在编译时就传入了。

  • 继承

    用一个特定的参数表示我们需要的功能，只用一个类（方法）来实现。

  • 硬编码

    从原来的基类派生出几个类。但是不使用override

返回值优化

RVO

  • 需要定义拷贝构造

  • 如果有多个return，返回值名称要一样

  • 编译器喜欢给未命名变量优化：

    计算性构造函数

临时对象

类型不匹配

构造函数传入的参数不匹配时，c++会进行类型转换。这一过程带来临时对象。

如果不希望这样，使用explicit 或者重载operaotr=()

按值传递

尽量不使用按值传递。当然，有时必须得传值，可以用拷贝构造等方式来优化。

按值返回

可以使用直接构造来规避这个问题（或者用RVO）

使用op=()或者+=()

重写op=()也可以优化。

内存管理

频繁使用new delete 会显著下降性能。因此如果对性能要求严格：

  • 使用栈内存

  • 预分配（像vector那样）

  • 自写一个内存管理器。

此处不再过多展开。

对于高效管理内存，还可以用引用计数。

内联

证明一个语言不是正则语言

  • \(L=\{a^nb^n|n\ge1\}\) 不是正则集

  • \(L=\{a^{i^2}|i\ge1\)的整数\(\}\) 不是正则语言

    假设当i>k时满足条件，\(\omega=a^{k^2}=\omega_1\omega_0\omega_2\) 。当\(i=2\)时，\(k^2\lt|\omega_1\omega_0\omega_2|\le k^2+k<(k+1)^2\) (多了一个\(\omega_0\)，最多多k个。)

  • \(L=\{www|w\in\{a,b\}*\)}$ 不是正则语言

L01: Basics of Information

Quantifying Information

信息的衡量 \[ I(x_i) = log_2{\frac{1}{p_i}}\\ p_i为概率 \]

Information Conveyed by Data

数据量的信息量 \[ I(data)=\log_2(\frac{1}{M/N})\\ N,M表示原来有N种选择，现在变成了M种 \]

Entropy熵

熵 \(H(x)\) 表示数据含有的信息量的期望，定义为: \[ H(X)=E(I(X)) = \sum_{i=1}^Np_i\cdot \log_2\frac{1}{p_i} \] 熵的含义是，平均来看，如果我们用少于熵的数据量发送信息，信息量可能不足（熵是我们要传递指定信息量的数据量下限）；另一方面，如果多于熵，则可能降低效率。

Encodings编码

如果各种选择出现的次数相同，我们通常用固定长度编码（Fixed-length Encodings）。

后面讲了数字编码（原码，补码Two's Complement)

对于出现概率不同的编码，我们用可变变长编码variable-length Encodings。可以用Huffman算法来构建。

Error Detection

汉明距离Hamming Distance

两个等长01串之间的不同之处。

比如

最多只发生一次错误的话：

要检测\(E\)个错误，汉明距离至少为\(E+1\)

要纠正\(E\)个错误，汉明距离至少为\(2E+1\)

L02: The Digital Abstraction

Using Voltages to Encode a Picture

此处用模拟电压来编码一张图片。但是编码的效果很差，因为编码的每一个阶段都存在大量的噪声。因此换用数字编码

Using Voltages Digitally

直接设定阈值不是一个好主意。因为在阈值附近时很难确认信号（或者说需要花费高精度的设备确认）是0或者1。

更好的尝试是：在中间设定一段“无效区”

Combinational Devices组合设备

需要满足以下四个点：

  • 有数字输入
  • 输出时数字信号
  • 对每组输入信号的值，都有一个确定的输出
  • 输入输入信号到产生信号有一个确定的传播时间。

A Combinational Digital System

  • 每个电路元件都是组合的
  • 每一个输入都有确定的输出
  • 电路没有有向环路

  • ....

发表于 2021-01-31

折腾了很久，写出来的博客可用性还是太差了。现阶段还是先用hexo比较方便。

插件

next主题

这里比较坑的是，hexo-next的主页好久没更新了（可能是因为维护的人换了），主页上链接是旧版的，在新的hexo下生成出来就只有模板。因此要用https://github.com/theme-next/hexo-theme-next 下的资源。

hexo-deployer-git

hexo-renderer-pandoc

用来渲染tex的。

配置参见Hexo+Next: 使用Latex公式。需要提前装好pandoc

修改配置文件是next主题里的_config.yml 。per_page 关了不会每页都渲染，只有加了mathjax:true的才会渲染。

图床

一开始想直接用hexo-asset-image。后面发现这个项目好久没更新了，并且实际用起来也不是那么方便，每个markdown都有一个asset文件夹。

专业图床有很多，比如七牛云，腾讯云COS，sm.ms

免费的图床sm.ms可以来存不重要图片。

国内的专业图床都要域名备案。七牛云可以试用一个月，我试了一下picgo+七牛云，感觉还是非常方便的。但是这个备案非常麻烦，而且目前我也没有长期使用的云服务器来备案，因此还是用github搭建图床吧。用gitee也可以，但是为了避免一些问题，还是选择比较稳定的github吧。github图床搭建只需要生成一个token，然后在picgo上面配置这个token即可。github加载图片还是比较慢的，如果要加速，还是需要CDN。当然github上面放图床不太好，以后再迁移到云服务器上吧。
